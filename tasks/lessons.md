# Lessons

## 2026-02-24
- OpenMyHealth의 핵심 가치는 "데이터 수집"이 아니라 "안전한 전달 과정"이다. 모든 기능은 최소전송 + 명시적 승인 UX를 먼저 만족해야 한다.
- ChatGPT UI와 제품 UI를 혼합하지 않는다. 사용자가 독립 보안 레이어로 인지할 수 있도록 Side Panel 중심으로 설계한다.
- 글로벌 확장성은 국가별 어댑터 인터페이스로 확보하고, 코어 파이프라인(저장/검색/승인/전송)은 국가 중립적으로 유지한다.
- 민감정보를 다루는 흐름에서는 전송 전 미리보기와 항목별 제외 기능을 필수로 제공한다.
- OAuth/토큰 하이재킹, 사용자 몰래 호출 같은 우회 전략은 제품 방향에서 제외한다. 신뢰 가능한 동의 기반 플로우만 제안/구현한다.
- 확장 E2E에서 입력창 주입 검증은 `toContainText`가 아니라 `toHaveValue`를 사용해야 실제 동작 검증이 된다.
- 잠금 해제 상태에서 인증 버튼 동작을 재사용하면 UX 혼란이 생긴다. 상태별 버튼 활성/비활성 로직을 명확히 분리한다.
- 소스 가이드는 Side Panel만으로는 부족할 수 있으므로, 페이지 내 오버레이(진행률+현재 단계 힌트)를 병행하면 완료율이 올라간다.
- CI에서 `playwright test`를 쓰는 스크립트는 최소 1개 테스트를 반드시 유지해야 한다. 테스트가 0개면 릴리즈 파이프라인이 즉시 실패한다.
- 크롬 확장 E2E를 CI에 넣을 때는 headful 제약을 고려해 `xvfb-run` 래퍼 스크립트를 제공해야 한다.
- 릴리즈 워크플로우와 동일한 경로를 로컬에서 반복 검증하려면 `release:dry-run` 같은 통합 스크립트를 유지해야 한다(아티팩트/체크섬까지 포함).
- 사이드패널 기반 제품은 "열림" 자체가 핵심 UX이므로 `openPanelOnActionClick` + `action.onClicked -> sidePanel.open`를 기본값으로 두고, 탭별 `setOptions(enabled)` 재적용을 `onStartup`까지 포함해 방어적으로 처리해야 한다.
- 확장 E2E가 단순 HTML 주입만 검증하면 실제 사용자 이슈(패널 enable/오픈) 누락이 생긴다. 도메인별 side panel option 검증 스펙을 별도로 유지해야 한다.
- 개발자 UX 요구(“UI를 보면서 디테일 조정”)가 강한 프로젝트는 초기부터 WXT/Plasmo 같은 dev 서버를 병행해야 한다. 수동 `chrome://extensions` 리로드 루프만으로는 반복 속도가 부족하다.
- 정적 UI 템플릿이 큰 경우 `static/*.html`을 단일 소스로 유지하고, dev 엔트리포인트에서 raw import + mount 후 앱 스크립트를 부트스트랩하면 릴리즈/개발 경로를 동시에 맞출 수 있다.
- 사이드패널은 "어디서 열리는지"가 UX 핵심이다. 허용 도메인을 최소화하고(`setOptions(enabled)`), 탭별 오픈을 위해 `openPanelOnActionClick` 자동 오픈은 끄고(`false`) `action.onClicked -> sidePanel.open({ tabId })`로 제어해야 비대상 탭에서 24/7 고정 패널 문제가 줄어든다. 또한 `onActivated/onUpdated`에서 `sidePanel.close`를 함께 호출해 잔존 패널을 정리한다.
- 승인 프리뷰 생성 후에는 해당 프리뷰 payload(IDs, redactionCount 포함)를 상태에 원본 그대로 보관해야 한다. 전송 시 별도 재조합하면 감사 로그와 실제 전달 컨텍스트가 불일치할 수 있다.
- 후보 체크박스 변경은 프리뷰를 즉시 무효화해야 한다. 선택 집합이 바뀐 뒤 이전 프리뷰를 재사용하면 승인 의미가 깨진다.
- AI 대상 탭 선택은 `활성 탭 우선`이 안전하다. 마지막 방문 AI 탭을 무조건 우선하면 사용자가 의도하지 않은 탭에 컨텍스트를 주입할 수 있다.
- 마이그레이션/리팩터링 이후에는 의존성 고아 패키지를 반드시 재점검해야 한다. 코드 참조가 없는 런타임/타입 패키지(`dayjs`, `jsbn` 등)는 즉시 제거해 번들·공급망 표면적을 줄인다.
- 민감정보 마스킹은 "보안 강화" 명목으로 임상 의미를 훼손하지 않도록 설계해야 한다. 날짜 전체 마스킹은 타임라인 이해를 깨므로 `생년월일/DOB`처럼 명확한 식별자 레이블 기반 규칙을 우선 적용한다.
- 타입/린트는 `src`만 보면 충분하지 않다. `entrypoints`를 검증 범위에 포함하지 않으면 실제 번들 진입점 회귀를 놓칠 수 있다.
