# Lessons

## 2026-02-23
- 검색 품질 개선 시 키워드/동의어/의학 용어를 코드에 하드코딩하지 않는다. 쿼리 확장과 정규화는 반드시 코퍼스/인덱스 기반으로만 수행한다.
- 사용자가 성능 요구를 변경하면(예: "30초 내 최적화" -> "e2e 제거"), 진행 중 최적화 전략을 즉시 중단하고 요구사항에 맞는 더 단순한 해결책으로 재계획한다.
- 테스트를 위한 전용 우회 코드(mock/fast-mode)는 최종 요구에서 제외되면 제품 코드에 남기지 않고 정리한다.
- 양자화 요구가 명시적으로 "q4만"이면 fallback을 두지 말고 실패를 즉시 드러내도록 강제 설정한다.
- 초기 모델 다운로드 최적화 요구가 나오면 원격 로딩 최적화보다 로컬 번들링(`public/models` + `allowRemoteModels=false`)을 우선 검토한다.
- 브라우저 임베딩 성능이 부족할 때는 단일 미세 튜닝보다 `청크 수 감소 + 배치 확대 + bulk upsert + 벡터 경량 저장`을 함께 적용해야 체감 개선이 크다.
- 성능 튜닝은 반드시 벤치 결과 기반으로 해야 하며, 공격적인 기본값(예: 큰 chunk/큰 batch)이 항상 빠른 것은 아니다.
- 구조화 JSON 검색 품질이 낮으면 path flatten보다 “레코드 단위 semantic formatter”를 먼저 도입해 문맥을 보존한다.
- 외부 레퍼런스(formatPHR 등)가 주어지면 직접 코드 확인 후 요약 중심 데이터 모델을 우선 적용하고, 키 경로 나열은 최소화한다.
- formatter 품질 작업은 앱 코드와 분리된 순수 모듈로 만들어야 실제 샘플 fixture 기반 유닛테스트를 안정적으로 유지할 수 있다.
- 민감 샘플 fixture를 저장할 때는 `sample1.json`/`phr/` 패턴을 피한 안전한 경로명(`tests/fixtures/anon/case_*.json`)을 사용해 사전 검사 스크립트와 충돌을 막는다.
- 하이브리드 검색 회귀를 막으려면 fallback 경로의 정렬 기준은 dense baseline과 동일한 raw similarity를 유지하고, lexical 신호는 선택성이 충분할 때만 게이팅해서 제한적으로 적용한다.
- 반복 실험 속도가 병목이면 임베딩 캐시(텍스트 해시 -> 벡터)를 먼저 도입해 실험 회전 시간을 줄이고, 그 위에서 게이팅/가중치 튜닝을 수행한다.
- 문서 임베딩 입력을 `text`에서 `searchText(메타 포함)`로 전환하면 summary/detail 계열 회복 효과가 크게 날 수 있으므로, intent별 편차를 함께 모니터링해야 한다.
- 쿼리 임베딩은 `raw+normalized`가 전역 성능을 높이지만 날짜 힌트 쿼리에는 `raw-only`가 더 안정적일 수 있어 조건부 입력 전략이 유효하다.
- 짧은 쿼리(토큰 수 2 이하)에서는 `raw+normalized` 결합보다 `raw-only`가 과확장 리스크를 줄일 수 있다.
- UI 쿼리 경로에서 사용하지 않는 fallback 정렬 계산은 즉시 제거해 지연을 줄인다(정확도 영향 없음).
- 긴 문서 커버리지를 늘리려는 head+tail 임베딩 입력은 직관과 달리 전역 성능을 쉽게 깨뜨릴 수 있으므로, 의도별(metric by intent) A/B 없이 기본값으로 채택하지 않는다.
- dense fallback에 lexical tie-break를 소량 추가하는 휴리스틱도 `summary_natural` 같은 intent를 쉽게 악화시킬 수 있어, fallback은 baseline dense 정렬을 우선 보존한다.
- `scripts/evaluate-retrieval.mjs`는 공용 임베딩 캐시 파일을 쓰므로 split 평가를 병렬 실행하지 않고 순차 실행해 캐시 경쟁을 피한다.
- 쿼리 확장은 `직접 매치 유무`보다 `lexicalCoverage` 기반 조건이 일반화에 더 유리했다. 낮은 coverage 구간에서만 Dice threshold/확장 개수를 동적으로 조정하면 global 및 colloquial 성능을 동시에 끌어올릴 수 있다.
- `doc.parts` 같은 구조화 메타를 코퍼스에서 추출한 토큰 집합(`partTermSet`)으로 활용하면 하드코딩 없이 질의 의도(부위/진료과)를 보강할 수 있다.
- 하이브리드 발동률은 5%->20%로 급증하면 split에 따라 과적합/회귀가 생길 수 있다. `part` 계열에는 별도 guard(최소 lexicalSupport)로 발동을 제어해야 한다.
- 보호 의도(`colloquial`, `part`)는 global 지표와 별도로 `hit@3/MRR` 비퇴화 게이트를 둬야 strict 평가에서 통과 가능하다.
- 날짜가 포함된 의료 질의는 월 힌트(`YYYY-MM`)를 별도 신호로 추출해 랭킹에 반영해야 한다. 임베딩만으로는 월 오매칭이 쉽게 발생한다.
- `월 컨텍스트` 같은 집계 섹션은 일반 진료/처방 문서보다 전역 후순위로 두는 것이 사용자 체감 정확도와 지표를 동시에 개선한다.
- 브라우저 재빌드 속도 병목은 모델 추론 자체보다 “반복 재임베딩”인 경우가 많아, IndexedDB 입력해시 기반 임베딩 캐시를 먼저 넣는 것이 효과가 크다.
- 실데이터 임베딩 벤치 결과로 배치 크기를 정해야 한다. 이번 환경에서는 `batch=12`가 `batch=8`보다 ms/chunk가 낮았다.
- strict 멀티 페르소나 리뷰에서 반복적으로 지적된 핵심은 평가 라벨 품질이다. 앵커 포함 기반 relevance만으로는 false-positive가 숨겨지므로, chunk-id/record-index 기준 정답 라벨이 필수다.
- summary 중심 포맷을 유지하더라도 `진료 상세/처방`에 월 단위 `진료과/질환`을 반복 주입하면 broad 질의에서 오탐이 크게 늘어난다. 섹션별 메타는 해당 섹션 고유 신호로 최소화해야 한다.
- `월 컨텍스트` 집계 문서는 설명용으로는 유용하지만 실제 검색 인덱스에 포함하면 top-rank 경쟁 노이즈가 될 수 있어, 필요 시 표시 전용으로 분리한다.
- strict 라벨이 `buildSemanticChunks` 산출물에 의존하므로 formatter 변경 후에는 반드시 `gen:retrieval-dataset`을 다시 실행해 평가 일관성을 맞춘다.
- 임베딩 성능 벤치는 sample 크기에 따라 최적 batch가 달라질 수 있다. 단일 수치만으로 고정하지 말고 2개 이상 샘플 크기로 재확인한다.
- `eval:retrieval`는 공용 캐시 파일을 쓰므로 split 동시 실행을 피하고 순차 실행을 기본 원칙으로 유지한다.
- WebGPU 미사용 환경의 체감 성능을 올리려면 WASM 스레드 수를 고정값 1로 두기보다 `crossOriginIsolated` 가능 여부를 보고 동적으로 설정해야 한다(격리 미충족 시 1 유지).
- 멀티 에이전트 루프를 돌릴 때는 `spawn_agent` thread limit을 먼저 확인하고, 한도 초과 시 즉시 대체 검증 루프(strict eval + 벤치 + 근거 리서치)로 전환한다.
- 짧은 구어체 의료 질의에서 성능이 안 나오면, 하드코딩 동의어 대신 코퍼스 DF 기반 broad-token pruning을 먼저 적용하라. `lexicalHitRatio`를 낮추면 hybrid gating이 정상화된다.
- `buildLexicalIndex`에서 `docFreq/docCount`를 노출하면 질의 토큰 품질 제어를 하드코딩 없이 구현할 수 있다.
- 평가 실행은 반드시 순차로 돌린다. 공용 캐시를 쓰는 스크립트를 병렬 실행하면 결과 재현성과 실행 시간 추적이 흔들린다.
- `normalized-only` 쿼리 임베딩 입력은 저특이도 질의에서 직관적으로 좋아 보여도 `summary_natural/part`를 동시에 악화시킬 수 있다. 반드시 split별 strict 재평가 후 채택한다.
- 랭킹 가중치 튜닝은 큰 폭보다 소폭(예: dense 0.78->0.74, lexical 0.18->0.22) 조정이 안전하다. hit@1을 유지하면서 hit@5/nDCG를 조금씩 누적 개선하는 전략이 유효했다.
- 요약 문서 granularity(레코드 묶음 크기)를 무작정 줄이면(`6 -> 4`) 문서 특이도는 올라가도 전역 랭킹 안정성이 떨어질 수 있다. 문서 수 증가가 곧 품질 향상은 아니다.
- 문서 임베딩 입력의 head-only 절단(`slice(0, limit)`)은 긴 의료 문서에서 핵심 항목 손실을 크게 만든다. header + 대표 레코드 라인 샘플링(compact input)이 split 전반에 더 안정적이었다.
- overlap 신호는 단순 토큰 개수보다 IDF 가중 비율이 훨씬 효과적이었다. 희귀 의학 용어/성분 토큰을 강조하면 `ingredient/prescription/detail` 성능이 동시에 개선된다.
- dense fallback에도 소량의 IDF overlap boost를 넣으면 hybrid 미발동 구간의 hit@1/MRR를 의미 있게 끌어올릴 수 있다(단, 가중치는 작게 유지).
- 구조 변경 후에는 가중치도 재튜닝해야 한다. 이번 케이스는 `dense=0.73`, `lexical=0.23`이 `0.74/0.22`보다 split 전반에서 우세했다.
- 저커버리지 질의에서 semantic expansion을 과도하게 켜면(`baseMatched=0` 외 coverage 조건까지 확대) hybrid 발동률이 급증하고 `part/colloquial` 안정성이 떨어질 수 있다.
- dense PRF blending은 이 데이터셋/파이프라인에서는 eval 일부 개선이 있어도 test 일반화가 약했다. PRF류는 기본값 채택 전 test split gate를 반드시 통과해야 한다.
- dense/lexical top 정합도(alignment)로 lexical 비중을 조절하는 휴리스틱은 직관적이지만, 현재 스케일에서는 노이즈를 줄이기보다 핵심 intent(ingredient/colloquial)를 악화시켰다.
- `part`/구어체 개선은 하이브리드 발동률을 늘리는 것보다 `partOverlap` 자체를 fallback/hybrid 모두에서 소폭 강화하는 접근이 더 안정적이었다.
- `partBoost`는 질의 강도(`partIntentStrength`)를 반영해 동적으로 주는 것이 고정 상수보다 안전했다.
- 캐시 공유 평가 스크립트는 병렬 실행을 피하고 반드시 순차 lock 결과를 남겨야 한다. 병렬 결과는 탐색용으로만 사용한다.
- 한국어 조사/어미 정규화는 직관적으로 좋아 보여도 test 일반화가 쉽게 깨질 수 있다. 형태 정규화 규칙은 eval 개선만으로 채택하지 말고 test MRR/nDCG 동시 개선을 필수 게이트로 둔다.
- EmbeddingGemma retrieval에서는 query/document 인코딩 프리픽스(`task: search result | query:`, `title: ... | text: ...`)를 적용하지 않으면 구어체/의도 분리 성능이 불안정해질 수 있다.
- `model_no_gather_q4`는 이름상 최적화처럼 보여도 현재 환경에서는 `model_q4`보다 크게 느릴 수 있으므로, 파일 변형 채택 전 동일 벤치(`ms/chunk`)를 반드시 실행한다.
- 공용 임베딩 캐시를 쓰는 평가를 실수로 병렬 실행했다면 해당 결과는 폐기하고, 동일 설정으로 eval→test 순차 재락을 다시 남긴다.
- ingredient처럼 동일 섹션 내 유사 청크 경쟁이 큰 경우, 문서 전체 overlap보다 `line-level max overlap`이 분별력에 더 효과적이다.
- line-level boost는 전역 적용보다 `처방/진료상세`처럼 긴 레코드형 섹션에만 소량 적용해야 summary 계열 회귀를 피할 수 있다.
- 실패 실험(v26)처럼 구조 조정이 test 전역 지표를 떨어뜨리면 즉시 롤백하고, 다음 후보는 동일 문제를 더 직접적으로 겨냥한 최소 변경(v27 line overlap)으로 전환한다.
- part/colloquial ambiguity를 줄이기 위한 file-diversify 조건 완화(v28)는 직관과 달리 part 지표를 악화시킬 수 있다. 다양성 휴리스틱은 보수적으로 유지한다.
- query 입력에 part-focus 라인을 추가(v29)해도 효과가 없을 수 있다. 성능 변화가 없으면 즉시 제거해 파이프라인 단순성을 유지한다.
- lineBoost 강도는 고정 상수보다 `lexicalSupport` 연동으로 미세 조정(v30)할 때 `hit@5/MRR/nDCG` 개선을 얻을 가능성이 높다.
- 영문 성분/화학식 질의는 raw query에서 추출한 verbatim phrase overlap을 별도 신호로 넣으면(하드코딩 없이) ingredient/prescription 분별력이 안정적으로 오른다(v31).
- 데이터셋 생성 규칙이 anchor 빈도 기반이면, 랭커도 focus token의 문서 내 TF를 약하게 반영해야 `part/colloquial`의 top-1 회복이 가능하다(v32).
- part-dominant 자연어 질의에서 generic 토큰(예: 기록/진료 문맥) 혼입은 순위를 불안정하게 만든다. 조건부로 part token 중심 축약을 적용하면 hit@1/MRR 개선 폭이 크다(v32).
- EmbeddingGemma ONNX는 `model_file_name` 선택에 따라 속도 차이가 크다. 기본 선택에 의존하지 말고 `model` 우선 고정 + `model_no_gather` 폴백 구조로 명시해야 한다(v17).
- embedding cache key에는 `modelId/dtype`만이 아니라 `modelFileName`도 포함해야 한다. 누락 시 다른 그래프 산출 벡터가 섞여 성능/재현성이 흔들린다.
- 배치 최적값은 synthetic/real 샘플에서 다르게 나올 수 있다. 고정 배치보다 device/model별 소규모 런타임 autotune + 로컬 캐시가 더 안전하다.
- part/colloquial 질의에서 focus TF를 `cap=6`으로 포화시키면 정답 분별 신호가 사라진다. part 질의는 높은 cap(예: 20)으로 유지해야 dataset 상위 후보를 안정적으로 올릴 수 있다(v33).
- part 질의에서는 overlap(존재 여부)보다 빈도(반복 강도) 신호가 더 중요하다. `partOverlap`만 강화하는 것보다 `focusFrequency` 계수 조정이 효과가 컸다.
- 보정 실험(v33b)처럼 계수를 너무 빠르게 완화하면 test `MRR/nDCG`가 즉시 떨어질 수 있다. 강한 후보(v33)를 baseline으로 두고 한 번에 작은 변화만 검증하는 것이 안전하다.
- `spawn_agent`가 thread limit으로 막히는 세션에서는 멀티 에이전트 루프를 고집하지 말고, strict eval 반복 + 공식문서/논문 근거 리서치로 의사결정 근거를 보강한다.
- 영문 성분/복합어 질의는 phrase 존재 여부만으로는 분별이 부족할 수 있다. 문서 내 phrase 반복 밀도(빈도/라인 수 정규화)를 함께 반영하면 ingredient `hit@1/MRR` 개선 폭이 크다(v34c).
- first build에서 embedding cache가 비어 있으면 per-chunk `cache get`은 순수 오버헤드다. cache store count=0인 빌드는 lookup을 생략하는 fast-path를 두는 것이 안전하고 효과적이다.
- 느린 환경에서 대용량 real benchmark는 실행 시간이 과도하게 길어질 수 있다. 먼저 “청크 1개 반복” quick benchmark로 batch 후보를 좁힌 뒤, 필요할 때만 full benchmark를 돌린다.
- part/colloquial 보정 신호를 오프라인 재랭킹 시뮬레이션으로 탐색할 때, `intent` 기반 적용 조건은 런타임에 재현 불가능하다. 실제 코드에선 `queryInfo`만으로 게이팅해야 하며, 시뮬레이션 기대치를 그대로 신뢰하면 과적합된다.
- part 밀도/순도 계열 신호는 일부 `hit@5`를 올려도 전역 `MRR/nDCG`를 쉽게 떨어뜨릴 수 있다. strict 기준에서 핵심 지표가 하락하면 즉시 비채택하고 롤백 락 결과를 남긴다.
- part 질의는 큰 리랭킹보다 미세 보정이 안전하다. `density(+), mixed-part penalty(-), compactness(+)`를 낮은 계수로 넣으면 `hit@1` 비열화 없이 `hit@3/MRR` 소폭 개선을 만들 수 있다(v36).
- 미세 개선이라도 strict eval/test 양쪽에서 전역 지표가 모두 비열화인지 확인해야 채택한다. 단일 split 개선만으로는 채택하지 않는다.
- verbatim phrase 추출을 영문 전용으로 두면 `detail/prescription`의 한국어+특수문자 항목명 질의를 놓친다. 패턴 기반 structured phrase 추출(날짜 제외)을 추가하면 하드코딩 없이도 큰 개선이 가능하다(v37).
- 구조화 phrase 확장 후에는 `detail_category/prescription` intent 지표를 별도로 확인해야 한다. 전역 상승이 보여도 특정 intent만 과도하게 당겨지지 않았는지 함께 점검한다.
- part 질의에서 hybrid를 일괄 비활성화하면 일부 broad 케이스는 좋아져도 구어체에서 역회귀가 생긴다. `partTokenRatio`만으로 끊지 말고 `lexicalHitRatio` 같은 spread 신호를 함께 써서 게이트를 좁혀야 한다(v39).
- 하이브리드 게이트 튜닝은 전역 수치만 보면 손실 원인이 숨겨진다. `vN -> vN+1` 비교 시 `queryId` 단위로 rank/rr 변화를 추적해 회귀 패턴을 먼저 고정해야 안정적으로 개선된다.
- 모호 part 자연어 질의에 파일 다양성 강제(diversify) 리랭크를 넣으면 직관과 달리 `hit@3/MRR`가 함께 떨어질 수 있다. 다양성은 기본값이 아니라 명확한 비즈니스 목적이 있을 때만 제한적으로 적용한다(v40 비채택).
- `queryInfo.lexicalCoverage`를 게이트에 추가하는 실험은 구분력이 좋아 보여도 실제로는 과도한 hybrid 비활성화로 이어질 수 있다. coverage 신호는 gating보다 가중치 미세조정(예: focusFrequency multiplier) 쪽이 더 안전했다(v41 비채택, v42 채택).
- 라벨 생성이 anchor 반복 빈도에 민감한 환경에서는 ambiguous part 질의군에서 `focusFrequency`를 조건부로 소폭 강화하는 것이 효과적이다. 전역 계수 상향보다 조건부(`partTokenRatio/baseTokenCount/lexicalHitRatio`) 가중치가 일반화 리스크가 낮다(v42).
- EmbeddingGemma는 MRL 차원 축소를 적극 탐색해야 한다. `512`가 무조건 최적이 아니고, 실제 strict 측정에서 `576`이 품질/속도 절충점으로 더 우수했다(v26).
- 차원 축소를 제품에 반영할 때는 임베딩 캐시 키에 `dim`을 반드시 포함해 기존 캐시와 혼용을 차단해야 한다.
- 차원 축소 효과 평가는 단일 지표로 결정하지 않는다. `hit@1/3/5 + MRR + nDCG`를 eval/test 모두 보고, 필요하면 `640/576/512`처럼 중간 차원까지 sweep해야 안정적으로 최적점을 찾을 수 있다.
- 576차원으로 차원 축소한 뒤에는 기존 결합 가중치(`dense/lexical`)를 그대로 두지 말고 다시 튜닝해야 한다. 작은 가중치 이동(`0.73/0.23 -> 0.71/0.25`)만으로도 eval/test에서 MRR/nDCG를 동시에 소폭 개선할 수 있었다(v27).
- near-tie를 `chunkIndex` 위치 prior로 직접 밀어붙이는 방식은 직관과 달리 `part/colloquial`뿐 아니라 전역 `hit@3/hit@5`를 악화시킬 수 있다. tie-break는 구조 신호 추가보다 먼저 결합 가중치 재튜닝으로 해결하는 편이 안전했다(v46 비채택).
- `colloquial` 계열은 전역 가중치를 크게 흔드는 것보다, `ambiguousPartNaturalQuery`로 좁힌 조건부 재가중(lexical↑, dense↓)이 더 안전하게 먹힌다. 이 방식은 `hit@1` 비열화 없이 `hit@3/MRR/nDCG`를 eval/test 모두에서 개선했다(v28).
- 실험 로그가 길어질수록 “모든 질의 공통 가중치”보다 “특정 실패 군집 전용 동적 가중치”가 효율적이다. 단, 동적 조건은 반드시 코퍼스 통계 기반으로 정의해 하드코딩 회귀를 피해야 한다.
- `ambiguous` 구간에서 dense fallback을 gap으로 강제하는 방식은 쉽게 전역 지표를 떨어뜨린다. 먼저 재가중(lexical/dense 비율) 스윕을 돌리고, fallback 강제는 마지막 수단으로 남기는 것이 안전했다(v29).
- `colloquial`은 top-1보다 top-3/5 개선 여지가 더 클 수 있다. 이 데이터셋에서는 `1.9/0.8` 재가중이 top-1 비열화 없이 top-5와 MRR/nDCG를 올리는 최적점이었다(v55).
- `chunkIndex` 기반 타이브레이크를 “근접 점수 구간”에만 제한해도 구어체 의도에서 전역적으로 악영향이 날 수 있다. 구조적 타이브레이크는 부분 성공 사례가 보여도 strict 전체 지표를 먼저 확인하고 즉시 롤백해야 한다(v59 비채택).
- ambiguous 재가중은 단발 튜닝보다 스윕이 필수다. v55(1.9/0.8)에서 멈추지 않고 `2.6/0.65`까지 밀어보니 eval/test 모두 `hit@3`와 `MRR/nDCG`가 추가 개선됐다(v60).
- ambiguous 조건 자체를 넓히는(baseToken/hitRatio 범위 확대) 시도는 지표가 그대로인 경우가 있다. 변화가 없는 실험은 즉시 폐기하고, 효과가 있는 축(재가중 강도)에 탐색 예산을 집중하는 것이 효율적이다.
- 임베딩 차원은 주기적으로 재검증해야 한다. 최근 파이프라인에서도 `576`이 `full/640/512`보다 strict 전역 안정성이 높아 기본값 유지가 타당했다.
- 섹션 친화도 보정은 강하게 넣으면 eval `hit@5`를 쉽게 훼손한다. `topSectionConfidence`가 높은 경우에만 아주 작은 불일치 패널티를 주는 방식이 더 안전했다(v61c).
- month mismatch 신호는 직관적으로 좋아 보여도 단독으로는 효과가 없을 수 있다. 무효 실험(v61b)처럼 변화가 없으면 즉시 폐기하고 효과가 있는 신호(섹션 불일치 미세 보정)로 전환해야 한다.
- 미세 보정 계수는 감으로 고정하지 말고 짧은 strict grid sweep으로 잠가야 한다. 이번에는 `c=0.62`, `w=0.007`이 `c=0.60`, `w=0.006` 대비 비열화 없이 nDCG를 추가로 올렸다(v62).
- month mismatch 계수(`known` penalty)는 이 데이터셋에서 `0.05~0.07` 범위 변화가 사실상 무효였다. 영향이 없는 축은 고정하고, 효과가 있는 축(섹션 보정)으로 탐색 예산을 집중해야 한다.
- file-level coherence boost는 직관적으로 cross-file 오탐을 줄일 것 같아도 실제로는 지표가 0 변화일 수 있다(v63). 무효 신호는 유지하지 말고 즉시 롤백해 랭커 복잡도를 억제한다.
- file cohesion 보정을 강하게 발동시키면 오히려 전역 성능이 악화될 수 있다(v64). `cross-file` 문제는 단일 휴리스틱으로 해결하려 하지 말고, 전역 비회귀가 깨지면 즉시 폐기해야 한다.
- verbatim phrase 가중을 동적으로 키우는 실험(v65)은 `hit@3/MRR` 일부 개선이 있어도 `eval hit@5`와 `test nDCG`를 떨어뜨릴 수 있다. 특정 의도 개선이 보여도 strict 전역 게이트를 우선한다.
- 결합 가중치(`dense/lexical`) 재튜닝(v66)과 section boost 스윕(v67)은 eval 상 이점이 보여도 test 일반화가 안 되거나 변화가 0일 수 있다. “보이는 개선”보다 test 게이트 통과를 우선한다.
- 구조화 문자열 매칭은 exact substring 하나로는 손실이 크다. `punctuation 제거 정규화 + includes` 기반의 `structuredPhraseOverlap`을 아주 작은 계수로 추가하면 strict 전역 비회귀를 지키면서 `test hit@3/MRR/nDCG`를 올릴 수 있었다(v68).
- structured phrase 보정은 강도가 핵심이다. `0.014/0.012`는 과강도(테스트 회귀), `0.004/0.003`은 eval hit@5 회귀, 최종 `0.003/0.002`만 안전했다.
- structured phrase 보정은 `safe zone` 미세 스윕이 유효했다. `0.0038/0.0026`까지 올려도 test 전 지표를 유지하면서 eval `hit@1/MRR/nDCG`를 추가 개선할 수 있었고(v69), 그 이상 강도는 재검증 전 채택하지 않는 것이 안전하다.
- EmbeddingGemma q4의 최적 batch는 환경별로 크게 다를 수 있다. 후보군을 `8/12/16`에 고정하면 최적점을 놓칠 수 있으므로(실측에서 `32`가 `12`보다 유의미하게 빠름), 장치별 후보 확장(`webgpu: 12~48`, `wasm/default: 8~32`) + 런타임 autotune이 필수다.
- long single-part 자연어 질의에서 lexical damp + dense recover 보정은 직관과 달리 `colloquial hit@1` 개선 없이 전역 `hit@3/hit@5`를 떨어뜨릴 수 있다(v70a). ambiguous 재가중 축은 조건 추가보다 strict 스윕 우선으로 다뤄야 한다.
- 임베딩 입력 길이 축소는 반드시 strict gate로 막아야 한다. `embedCharLimit=750`는 빌드 속도 기대와 달리 retrieval 품질이 크게 무너졌고(`MRR/nDCG` 급락), 단순 길이 절단은 안전한 최적화가 아니다.
- 1차 출처 기준으로 EmbeddingGemma는 task prefix와 MRL(dim truncation) 사용을 전제로 설계되어 있다. 현재 파이프라인처럼 `query/document prefix + dim=576` 조합을 유지하면서 다른 축을 튜닝하는 것이 안전하다.
- ONNX Runtime Web/Transformers.js 가이드상 브라우저 추론 성능의 핵심은 backend 선택(WebGPU 우선)과 dtype/배치 튜닝이다. CPU/WASM 구간은 고정 배치보다 장치별 autotune이 필수다.
- 운영 시나리오가 “고객별 단발성 로컬 처리”라면 단순 임베딩 캐시는 측정/체감을 왜곡할 수 있다. 이런 경우 문서/쿼리 임베딩 캐시는 기본 비활성화로 두고 순수 추론 성능 기준으로 튜닝한다.
